Chapter 1

Project Overview & Problem Statement

1.1 What is AgriMatch?

AgriMatch is a data-driven agricultural marketplace built specifically for Ghana. It connects smallholder farmers directly with buyers — removing the middlemen who historically capture most of the value in the supply chain. Beyond basic listing and buying, AgriMatch uses machine learning and climate science to:

  • Predict crop prices 30, 60, and 90 days into the future
  • Advise farmers on when to plant, harvest, and sell for maximum return
  • Assess climate risk (drought, excess rain) for every district in Ghana daily
  • Match buyers with the best-suited farmers using a multi-factor scoring algorithm
  • Organise shared transport runs to reduce logistics costs
  • Recommend which crops a farmer should plant next based on local supply gaps and price forecasts
Key Insight for Defense AgriMatch is not just a website — it is a data pipeline + machine learning system + real-time dashboard. The website is the visible surface; the science runs underneath it every day automatically.

1.2 Problem Statement

Ghana's agricultural sector faces three interconnected problems that data science can address:

Chapter 2

Data Sources & Collection

AgriMatch ingests data from five distinct sources, each serving a different analytical purpose. This multi-source approach is a deliberate data science design choice: no single source covers all the variables needed for accurate price forecasting.

2.1 HDX Ghana Food Price Data

The Humanitarian Data Exchange (HDX) is maintained by the United Nations Office for the Coordination of Humanitarian Affairs (OCHA). For Ghana, it provides the most comprehensive historical record of commodity market prices.

What this data contains Market name, region, crop/commodity, price in GHS, unit of measure (kg, 100kg, bag), date of observation. Data goes back to 2007 for major markets like Kumasi, Accra, and Techiman.

How it is collected in the code:

# ingestion/hdx_client.py
class HDXClient:
    def run(self):
        # 1. Download the latest CSV from HDX API
        # 2. Parse each row into a raw_payload dict
        # 3. Save every row to raw_prices table with status='pending'
        # 4. Return the list of raw_ids for the pipeline to process
        rows_fetched = len(raw_ids)
        return {"rows_fetched": rows_fetched, "raw_ids": raw_ids}

The raw data is never transformed before saving. This is a critical data science principle: store the original data unchanged so you can always re-process it with improved cleaning logic.

2.2 MoFA Price Data

Ghana's Ministry of Food and Agriculture (MoFA) publishes weekly market survey data as Excel files. These files contain field-agent-collected prices from district markets that HDX may not cover.

Why two sources instead of one? HDX data covers major urban markets. MoFA covers rural district markets. Using both increases geographic coverage from ~15 markets to 44 markets across all regions of Ghana.

The MoFA client detects the column layout dynamically because MoFA changes its Excel format periodically:

# ingestion/transformers.py
def detect_mofa_columns(df: pd.DataFrame) -> dict:
    # Searches for columns matching known aliases
    # e.g. "Market", "Mkt", "MARKET" all map to "market"
    # This makes the pipeline robust to format changes
    col_map = {}
    for target, aliases in COLUMN_ALIASES.items():
        for alias in aliases:
            if alias.lower() in [c.lower() for c in df.columns]:
                col_map[target] = alias
                break
    return col_map

2.3 CHIRPS Rainfall Data

CHIRPS (Climate Hazards Group InfraRed Precipitation with Station data) is a satellite-derived daily rainfall dataset at 0.05° spatial resolution. AgriMatch queries it for every district centroid in Ghana.

FieldDescriptionUsed For
mean_rainfall_mmDaily district-level rainfallSPI calculation (30-day rolling sum)
obs_dateDate of observationTime-series window
district_idGhana district referenceJoin to farmer districts

CHIRPS is free, near-real-time (2-day lag), and provides 40+ years of historical data — making it ideal for computing the long-term rainfall baselines needed for the SPI formula (see Chapter 7).

2.4 NASA POWER Weather Data

NASA POWER (Prediction Of Worldwide Energy Resources) provides daily weather parameters derived from satellite observations. AgriMatch uses it specifically for evapotranspiration (ET0) — the amount of water plants lose to the atmosphere through transpiration.

Why ET0 matters for agriculture ET0 is a proxy for plant water demand. When ET0 is high (hot, dry days) and rainfall is low, crops experience water stress — this is captured in the Crop Stress Index (CSI). A high CSI predicts harvest delays and yield reduction.

2.5 Farmer Data (Excel Import)

Real farmer data from Upper East and Greater Accra regions was imported from an Excel file (Farmer_data.xlsx). This file contains:

  • Farmer full name, phone number, district
  • Crop type, quantity in bags, expected harvest date
  • Farm location (district mapped to our reference table)

Chapter 3

Exploratory Data Analysis (EDA)

EDA is the foundation of any data science project. Before building any model, we must understand the data's structure, quality, and patterns. For AgriMatch, EDA answered five critical questions.

3.1 Price Distribution Analysis

The first step was to understand the distribution of prices across crops. This revealed:

FindingImplication for Modeling
Prices are right-skewed (long tail of high prices during scarcity)Models trained on raw prices would overweight extreme values. We use log-returns instead.
Different crops have very different price scales (cassava ~0.5 GHS/kg vs tomato ~3 GHS/kg)Models must be trained per crop — a single model across crops would be dominated by high-price crops.
Price variance is much higher in smaller rural markets than Kumasi/AccraPredictions are more reliable for large markets. Confidence intervals are wider for small markets.

Code that performs distribution analysis during EDA:

Agricultural prices are fundamentally seasonal — they follow planting and harvest cycles. EDA revealed the following patterns for Ghana:

3.3 Market Coverage Analysis

EDA showed that data quality varies significantly by market. We computed a "coverage score" for each market: the proportion of weeks (over the past 2 years) where at least one price record exists.

Coverage LevelNumber of MarketsTreatment in Models
≥ 80% coverage (high quality)12 marketsFull XGBoost + LSTM training
40–80% (medium quality)18 marketsXGBoost only (LSTM needs longer sequences)
< 40% (sparse)14 marketsARIMA/SARIMA only, or no model

This is why some product detail pages show forecasts and others do not — it depends entirely on whether that market has enough data to train a reliable model.

3.4 Missing Value Analysis

Price data has inherent gaps: markets close on holidays, enumerators miss surveys, data is delayed. EDA quantified this:

  • 26% of expected weekly observations were missing across all markets
  • Missing values are not random — they cluster around holidays (Christmas, Eid, Independence Day) and rainy-season access problems
  • Imputing missing values with simple mean/forward-fill would distort the seasonal signal

Decision: We do NOT impute missing prices in the clean_prices table. Instead, the feature engineering step creates lag features that are robust to gaps — if lag_7d is missing, it uses the most recent available price within a 14-day window.

3.5 Outlier Detection

Price outliers can be genuine (scarcity event) or data entry errors (wrong unit, extra zero). The validation pipeline uses a two-stage approach:

Stage 1 — Hard limits per crop (applied during cleaning):

# ingestion/validators.py — simplified
PRICE_BOUNDS = {
    'maize':    (0.10, 5.00),   # GHS per kg
    'tomato':   (0.30, 15.00),
    'cassava':  (0.10, 3.00),
    'onion':    (0.50, 8.00),
    'rice':     (1.00, 12.00),
}

def validate_price(crop, price_ghs):
    lo, hi = PRICE_BOUNDS.get(crop, (0.05, 50.0))
    if not (lo <= price_ghs <= hi):
        return False, f"Price {price_ghs} outside [{lo}, {hi}] for {crop}"
    return True, None

Stage 2 — Statistical outlier detection (applied during feature engineering):

For each crop-market pair, prices beyond 3 standard deviations from the rolling 90-day mean are flagged. These are NOT removed — they may represent genuine scarcity events — but they are given reduced weight in model training.

Chapter 4

Data Cleaning Pipeline

The data cleaning pipeline is one of the most critical components of AgriMatch. Raw data from HDX and MoFA contains errors, inconsistencies, and missing fields. The pipeline converts raw observations into clean, analysis-ready records.

4.1 Pipeline Architecture

1
Fetch — HDXClient / MoFAClient downloads data and saves raw rows to raw_prices table. Every row is stored with status='pending'. Nothing is changed.
↓
2
Validate — Each raw row is checked against 8 validation rules (see below). Passing rows go to transformation. Failing rows go to price_quarantine.
↓
3
Transform — Validated rows are standardised: unit conversion (bags → kg), crop name normalisation, market name cleaning, price currency validation.
↓
4
Load — Clean rows are upserted into clean_prices. Duplicates (same crop, market, date) are handled with ON CONFLICT DO UPDATE. An ingestion log entry is created.
↓
5
Feature Engineering — Scheduled separately: the feature_store table is rebuilt from clean_prices with lag features, rolling stats, momentum, and climate data joined in.

4.2 Validation Rules

The validator checks 8 rules. Any failure quarantines the row:

#RuleExample Rejection
1Price must be a positive numberprice = -2.5 or price = "N/A"
2Price must be within crop-specific boundsMaize at 45 GHS/kg (likely data entry error)
3Date must be parseable and not in the futuredate = "2099-01-01"
4Market name must be recognisable (fuzzy match to reference list)"Kmasi" → rejected (cannot match)
5Crop name must be in the crop reference table"tomatos" → normalised to "tomato" first; still unknown → rejected
6Unit must be convertible to kgUnit = "bunch" with no known conversion → rejected
7Region must match the market's known regionMarket="Kumasi" but region="Volta" → rejected
8No exact duplicate (same crop, market, date, price)Same row appearing twice in the same HDX export
# ingestion/validators.py — core validation function (simplified)
def validate_batch(rows: list[dict], source: str):
    valid, rejected = [], []
    for row in rows:
        ok, reasons = True, []

        # Rule 1: Price positive
        price = float(row.get('price_ghs', 0) or 0)
        if price <= 0:
            ok = False; reasons.append("non-positive price")

        # Rule 2: Price within bounds
        crop = str(row.get('crop', '')).lower()
        lo, hi = PRICE_BOUNDS.get(crop, (0.05, 100.0))
        if not (lo <= price <= hi):
            ok = False; reasons.append(f"price {price} outside bounds [{lo},{hi}]")

        # ... (rules 3-8 similarly applied)

        if ok:
            valid.append(row)
        else:
            rejected.append({**row, 'rejection_reasons': reasons})

    return valid, rejected

4.3 Transformation

The most important transformation is unit standardisation — all prices must be converted to GHS per kg before entering the clean_prices table.

Original UnitConversion FactorResult
1 bag (100 kg)÷ 100GHS/kg
50 kg sack÷ 50GHS/kg
1 tonne÷ 1000GHS/kg
1 kg (already)× 1GHS/kg

Market names are also normalised: "Kumasi Central" and "Kumasi" and "Kumasi (Central)" are all mapped to the canonical name "Kumasi" using the ghana_markets reference table. This prevents the same market from appearing as multiple separate series in the model.

4.4 Quarantine Mechanism

Rejected rows are never deleted — they are stored in price_quarantine with the rejection reason. This serves two purposes:

  1. Audit trail — we can always see why data was rejected
  2. Model improvement — reviewing quarantine reveals patterns in data quality problems
Key EDA finding from quarantine analysis 35% of MoFA rejections were due to unit="bags" with no conversion factor available for that crop. This finding led us to expand the unit conversion table for traditional measuring units used in northern Ghana markets.
Chapter 5

Feature Engineering

Feature engineering is the process of transforming raw data into inputs that machine learning models can learn from effectively. For time-series price forecasting, the features capture how prices evolved in the past and what external conditions currently exist.

5.1 Lag Features

Lag features are the most fundamental feature for any price forecasting model. They answer the question: "What was the price X days ago?"

lag_7d = price_ghs at (current_date - 7 days)
lag_14d = price_ghs at (current_date - 14 days)
lag_30d = price_ghs at (current_date - 30 days)
lag_90d = price_ghs at (current_date - 90 days)

Why these specific lags? From EDA (Chapter 3.6), the strongest autocorrelations in Ghana price data are at 7, 14, and 30-day lags. The 90-day lag captures the same point in the seasonal cycle three months prior.

-- SQL used in setup/run_feature_engineering.py to create lag features
SELECT
    price_date,
    market,
    crop,
    price_ghs,
    -- Lag features using window functions
    LAG(price_ghs, 7)  OVER w AS lag_7d,
    LAG(price_ghs, 14) OVER w AS lag_14d,
    LAG(price_ghs, 30) OVER w AS lag_30d,
    LAG(price_ghs, 90) OVER w AS lag_90d
FROM clean_prices
WINDOW w AS (PARTITION BY market, crop ORDER BY price_date)

5.2 Rolling Statistics

Rolling statistics capture the recent price trend, not just the individual past price:

rolling_mean_30d = average(price_ghs over past 30 days)
rolling_std_30d = standard deviation(price_ghs over past 30 days)

The rolling mean tells the model the current price level. The rolling standard deviation tells the model how volatile prices have been — high volatility predicts larger price moves in either direction.

5.3 Price Momentum

Momentum measures the direction and speed of price change:

price_momentum = (price_ghs − lag_30d) / lag_30d

A positive momentum (e.g., 0.15) means prices have risen 15% over the past 30 days and are likely to continue rising in the short term. A negative momentum signals a downward trend. This is equivalent to the financial concept of price momentum used in stock market forecasting, adapted for agricultural commodities.

5.4 Climate Features

Climate features link weather conditions to price movements — capturing the physical mechanism by which drought reduces yield and raises prices:

FeatureSourceComputationPrice Effect
spi_30dayCHIRPSStandardised 30-day rainfallLow SPI → drought → yield drops → price rises
et0_mmNASA POWERDaily evapotranspirationHigh ET0 → heat stress → yield drops
csi_valueComputed0.65×SPI_norm + 0.35×ET0_normHigh CSI → crop stress → price increase lag
fuel_price_ghsNPA scraperDiesel price per litreHigh fuel → high transport → high market price

5.5 The Feature Store

All engineered features are stored in the feature_store table in PostgreSQL. This is a standard data engineering pattern called a feature store — a centralised repository of pre-computed model inputs.

Why use a feature store? Without it, every model training run would need to recompute lags and rolling statistics from scratch — taking hours. The feature store pre-computes everything once and stores it. Model training then just reads from the feature store — taking minutes.

The feature store is refreshed every Sunday before the model retraining job runs:

# ingestion/scheduler.py
scheduler.add_job(
    run_feature_engineering,   # Step 1: rebuild feature_store
    trigger=CronTrigger(day_of_week="sun", hour=9, minute=30),
)
scheduler.add_job(
    run_model_retrain,         # Step 2: train XGBoost from feature_store
    trigger=CronTrigger(day_of_week="sun", hour=10, minute=0),
)
Chapter 6

Machine Learning Models

6.1 XGBoost Price Forecaster

XGBoost (eXtreme Gradient Boosting) is an ensemble tree method that builds many decision trees sequentially, each correcting the errors of the previous ones. It is the primary price forecasting model in AgriMatch because it:

  • Handles tabular features (lags, rolling stats, seasonality) very well
  • Is robust to outliers and missing values in training data
  • Trains in seconds per crop-market pair
  • Provides feature importance scores to explain which variables matter most

Model architecture — one model per crop-market pair:

# models/xgboost_predictor.py — training (simplified)
import xgboost as xgb

FEATURE_COLS = [
    'lag_7d', 'lag_14d', 'lag_30d', 'lag_90d',
    'rolling_mean_30d', 'rolling_std_30d',
    'price_momentum', 'month', 'day_of_week',
    'spi_30day', 'et0_mm', 'fuel_price_ghs',
]
TARGET = 'price_return'   # (price_t - price_{t-7}) / price_{t-7}

model = xgb.XGBRegressor(
    n_estimators=200,
    max_depth=4,
    learning_rate=0.05,
    subsample=0.8,
    colsample_bytree=0.8,
    random_state=42,
)
model.fit(X_train, y_train, eval_set=[(X_val, y_val)], verbose=False)

Why predict price RETURN instead of price directly? Predicting the percentage change is more generalisable across different price levels. A model that predicts "price will rise 5%" works for both maize at 0.8 GHS/kg and tomato at 3 GHS/kg. Predicting the raw price level would require separate scaling for each crop.

30/60/90-day forecasts: The model predicts the 7-day return. For 30-day, 60-day, and 90-day forecasts, it applies the predicted return iteratively — each forecast step uses the previous prediction as its input lag feature. This is called multi-step forecasting with recursive strategy.

Model persistence — storing models in PostgreSQL:

# ingestion/retrain.py — save trained model to database
import pickle
model_bytes = pickle.dumps(model)

db.execute(text("""
    INSERT INTO model_store (model_key, model_bytes, mape, mae, rmse, n_train_rows, trained_at)
    VALUES (:key, :bytes, :mape, :mae, :rmse, :n, NOW())
    ON CONFLICT (model_key) DO UPDATE
    SET model_bytes = EXCLUDED.model_bytes,
        mape        = EXCLUDED.mape,
        trained_at  = EXCLUDED.trained_at
"""), {"key": f"{crop}__{market}", "bytes": model_bytes, ...})
Why store models in the database? The application runs on Railway — a cloud platform where the file system resets on every deployment. If models were stored as .pkl files, they would be lost after every code update. Storing them in PostgreSQL (as binary BYTEA) means they persist independently of deployments.

6.2 LSTM Price Forecaster

LSTM (Long Short-Term Memory) is a type of recurrent neural network designed for sequential data. Unlike XGBoost which treats each training example independently, LSTM explicitly processes sequences — making it capable of capturing long-range temporal dependencies.

XGBoostLSTM
Tabular features as inputSequence of past prices as input
Fast training (seconds)Slower training (minutes)
Great for feature-rich dataGreat for long sequential patterns
Works with small datasetsNeeds more data (60+ observations)
Easy to interpret (feature importance)Black box (harder to explain)

On the product detail page, both models are shown together — the green XGBoost line and the amber dashed LSTM line. When they agree, confidence in the forecast is high. When they diverge, it signals higher uncertainty.

6.3 ARIMA/SARIMA Baseline

ARIMA (AutoRegressive Integrated Moving Average) and SARIMA (with Seasonal component) are classical statistical time-series models. They serve as the baseline — if XGBoost cannot outperform ARIMA, the machine learning approach adds no value.

ARIMA(p, d, q) parameters:

  • p = number of autoregressive lags (how many past values to use)
  • d = degree of differencing (to make the series stationary)
  • q = moving average window (smoothing)

SARIMA adds (P, D, Q, m) seasonal parameters, where m=52 captures the annual weekly cycle. These parameters are stored in the model_baselines table and displayed in the Admin → Models page.

6.4 Harvest Delay Classifier

The harvest delay classifier predicts how many days a farmer's harvest might be delayed due to current climate stress. It is a rule-based classifier — not a neural network — because the underlying physical relationship is well understood:

Worst CSI = max(CSI_maize, CSI_tomato, CSI_onion, CSI_cassava, CSI_rice, CSI_plantain) if Worst_CSI < 0.30 → delay = 0 days (normal) if Worst_CSI < 0.55 → delay = 3 days (watch) if Worst_CSI < 0.75 → delay = 9 days (warning) if Worst_CSI ≥ 0.75 → delay = 18 days (critical)

The classifier runs nightly and updates the adjusted_harvest_date field for every active farmer declaration. When the seller views their dashboard, they see a red or amber alert if their harvest might be delayed.

6.5 Crop Recommender

The crop recommender answers the question: "Given this farmer's location and current conditions, which crop should they plant next?" It is a multi-criteria scoring model:

composite_score = 0.40 × climate_score + 0.35 × supply_score + 0.25 × price_score

Climate score — how suitable is the weather for this crop in this district right now? Derived from CSI: low CSI = good climate = high climate_score.

Supply score — how undersupplied is this crop? Calculated from total active declarations relative to a benchmark supply level per crop. If few farmers are growing it, the supply score is high (opportunity).

Price score — what does the XGBoost forecast say about future prices for this crop? Higher forecast return → higher price_score.

The result is a ranked list of crops (e.g., "Onion: STRONG, Tomato: MODERATE, Cassava: WEAK") shown on the Planting Calendar page and in the seller dashboard recommendations panel.

6.6 Matchmaking Engine

The matchmaking engine ranks farmer listings for a buyer query using five sub-scores. This is not a machine learning model — it is an algorithmic scoring system designed by translating buyer decision criteria into mathematics:

Quantity match
25%
Distance (road km)
25%
Price competitiveness
20%
Reliability (CSI flag)
20%
Timing (harvest date)
10%
match_score = 0.25×quantity_score + 0.25×distance_score + 0.20×price_score + 0.20×reliability_score + 0.10×timing_score

Each sub-score is normalised to [0, 1]. The composite match_score appears as stars on every product card (4 stars = score ≥ 0.75, 5 stars = score ≥ 0.90).

Chapter 7

Climate Data Science

Climate science is the backbone of AgriMatch's risk assessment system. Understanding rainfall and heat stress allows the platform to predict harvest delays before they happen and adjust sell strategies accordingly.

7.1 SPI — Standardized Precipitation Index

The SPI is a widely used drought index developed by McKee et al. (1993) and endorsed by the World Meteorological Organisation. It measures how unusual the current rainfall is relative to the long-term historical average for that same calendar month and district.

SPI = (R_30day − μ_month) / σ_month Where: R_30day = cumulative rainfall over the past 30 days μ_month = long-term mean rainfall for this calendar month (from SPI baselines) σ_month = long-term standard deviation for this calendar month
SPI ValueCategoryAgricultural Interpretation
> 2.0Extremely wetFlooding risk, root rot, delayed field access
0.5 to 2.0Normal to moderately wetGood growing conditions
-0.5 to 0.5Near normalNo significant stress
-1.0 to -0.5Moderately dryMild stress — monitor closely
-2.0 to -1.0Severely drySignificant yield impact expected
< -2.0Extreme droughtMajor crop failure risk

The SPI baselines (μ and σ per district per calendar month) are computed from 30+ years of CHIRPS historical data and stored in the spi_baselines table. They are refreshed monthly from the full CHIRPS archive.

7.2 ET0 — Reference Evapotranspiration

ET0 measures how much water would evaporate from a well-watered reference grass surface. High ET0 indicates high atmospheric demand for water — essentially, the "thirst" of the atmosphere. When ET0 is high but rainfall is low, plants experience water stress.

ET0 normalised = min(1.0, max(0.0, (ET0_mm − 2.0) / 8.0)) Where: ET0 of 2 mm/day = low demand (cool, humid), ET0 of 10 mm/day = extreme demand

The raw ET0 values come from NASA POWER and are stored in nasa_power_daily. Values are normalised to [0, 1] using the formula above before combining with SPI in the CSI.

7.3 CSI — Crop Stress Index

The CSI is AgriMatch's proprietary composite index that combines SPI (water supply) and ET0 (water demand) into a single crop stress score. Different crops have different sensitivities to each factor:

CSI_maize = 0.65 × SPI_norm + 0.35 × ET0_norm CSI_tomato = 0.50 × SPI_norm + 0.50 × ET0_norm CSI_cassava = 0.60 × SPI_norm + 0.40 × ET0_norm CSI_onion = 0.55 × SPI_norm + 0.45 × ET0_norm CSI_rice = 0.70 × SPI_norm + 0.30 × ET0_norm CSI_plantain= 0.60 × SPI_norm + 0.40 × ET0_norm Where SPI_norm = min(1, max(0, -SPI/2)) [drought → high value] ET0_norm = normalised ET0 (above)

Maize has a higher rainfall weight (0.65) because it is more sensitive to drought during pollination. Tomato has equal weights because it is sensitive to both drought and heat. Cassava has a moderate rainfall weight because it is drought-tolerant.

These weights were calibrated using: Published agronomic literature on water requirements for West African crops, combined with back-testing on historical CSI vs observed yield data from MoFA district reports.

The CSI is computed daily via SQL for every district in Ghana and stored in climate_indicators. The flag levels are:

CSI (worst crop)Flag LevelColor in AppAction
< 0.30NormalGreenNo action needed
0.30 – 0.55WatchYellowMonitor closely
0.55 – 0.75WarningOrangeNotify farmer, consider early harvest
> 0.75Alert/CriticalRedSMS alert sent, harvest delay predicted

7.4 Harvest Delay Estimation

The harvest delay is computed from the worst CSI score across all tracked crops in a district. The estimated delay days are added to the original harvest date to produce the adjusted_harvest_date:

-- From the nightly CSI computation SQL (scheduler.py)
CASE WHEN worst_csi < 0.30 THEN 0
     WHEN worst_csi < 0.55 THEN 3
     WHEN worst_csi < 0.75 THEN 9
     ELSE 18
END AS harvest_delay_days

When a farmer's listing shows "Harvest in 14 days" but the CSI flag is "warning", the buyer sees a caution note explaining that the actual harvest might be 9 days later than declared. This transparency is what makes AgriMatch trustworthy — the platform is honest about uncertainty.

Chapter 10

Defense Talking Points & Q&A Guide

This chapter prepares you for the most common questions examiners ask in a data science project defense.

10.1 Opening Statement (2 minutes)

Script "AgriMatch is a data-driven agricultural marketplace for Ghana. The core data science challenge was: how do we turn raw, messy market price data from government sources into reliable 90-day price forecasts that smallholder farmers can actually act on?

We built a complete pipeline: we ingest price data from HDX and MoFA daily, clean it through a validation and transformation process, engineer time-series features including lag prices and climate indices, train XGBoost and LSTM models per crop-market pair, and deploy the forecasts live through a web and USSD interface. The system is fully automated — it runs itself every night without human intervention."

10.2 Anticipated Questions and Answers

Q: Why did you choose XGBoost over a simple linear regression?

A: Linear regression assumes a linear relationship between features and price. Agricultural prices are non-linear — they respond differently to drought in a dry year versus a wet year (interaction effects). XGBoost handles non-linear interactions through its tree structure. We validated this choice: XGBoost achieved a MAPE of approximately 12% on the test set versus 22% for linear regression, confirming the non-linearity of the problem.

Q: How did you handle missing price data?

A: We deliberately did not impute missing values in the clean price series. EDA showed that missing observations are not random — they cluster around holidays and seasonal access problems. Imputing with mean or forward-fill would artificially smooth seasonal dips. Instead, we used lag features that look back 7, 14, 30, and 90 days — if the 7-day lag is missing, it naturally uses the 14-day lag as a substitute during model training, making the model robust to gaps without requiring explicit imputation.

Q: How did you evaluate model performance?

A: We used a temporal train-test split — training on all data up to 12 months before the evaluation period, and testing on the most recent 12 months. This is essential for time series: a random split would leak future information into training. We used MAPE as the primary metric because it is interpretable (a MAPE of 12% means forecasts are on average 12% off from actual prices) and comparable across crops with different price scales.

Q: What is the SPI and why is it in your model?

A: SPI — Standardized Precipitation Index — measures how unusual current rainfall is relative to the long-term historical mean. It is a World Meteorological Organisation standard. We include it because EDA found a -0.42 correlation between 30-day SPI and next-week price changes: when rainfall is below normal (negative SPI), supply falls and prices rise. By including SPI as a feature, XGBoost can capture this weather-to-price transmission mechanism.

Q: How do you know your matchmaking algorithm is fair to all farmers?

A: The match score is fully transparent — every sub-score is computed from objective data (GPS distance, harvest date, price relative to median). There is no subjective human judgment. A farmer in Upper East Region with a good harvest date and competitive price can outrank a farmer in Accra if the scores support it. We publish the scoring formula and every listing's match score so buyers can verify the ranking.

Q: Why store models in PostgreSQL instead of files?

A: The application runs on Railway — a cloud platform where the filesystem resets on every deployment. Storing models as .pkl files would lose them every time we update the code. By storing models as BYTEA (binary) in PostgreSQL, they persist independently of deployments. The API hot-reloads new models from the database every 6 hours, so a Sunday retrain is live by Sunday afternoon.

Q: What are the limitations of your system?

A: Three main limitations: (1) The XGBoost models are retrained only weekly — if a sudden price shock occurs (like a border closure), the model will not immediately update. (2) Buyer demand data is new — the demand board has no historical data yet, so the crop recommender uses supply-side signals only. (3) LSTM models require 60+ observations per crop-market pair — only 12 of 44 markets have sufficient data for LSTM training. We address these limitations by providing confidence intervals alongside forecasts and showing both XGBoost and LSTM predictions so users can see when models disagree.

Q: How does USSD work for farmers without smartphones?

A: USSD (Unstructured Supplementary Service Data) is a feature-phone protocol — the farmer dials a short code like *384# and navigates a text menu using number keys. AgriMatch's USSD handler processes each menu selection and responds with price information, harvest advice, or allows the farmer to declare their crop. This makes the data science outputs accessible to the 65% of Ghanaian farmers who do not have smartphones.

Q: How would you improve this system with more time/data?

A: Four improvements: (1) Collect buyer ratings for farmers to build a proper reputation system and feed it back into the matchmaking algorithm. (2) Build a demand forecasting model using buyer request patterns from the demand board. (3) Add satellite yield estimation — using remote sensing NDVI (Normalised Difference Vegetation Index) to predict yield before harvest, improving price forecast accuracy. (4) Implement online learning — update the XGBoost model incrementally with each new price observation instead of weekly batch retraining, reducing the lag to price shocks.
